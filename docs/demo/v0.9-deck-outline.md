## 4. 技术路线：前后端分离让系统可扩展、可观察

**页面标题**：我选择了 React + FastAPI 的前后端分离架构

**为什么这样选**：

- 分离 UI 状态与分析执行，避免模型长任务阻塞交互层。
- 前后端通过 REST + WebSocket 建立清晰协议，便于测试和定位故障。
- 前端、后端和 Agent runtime 可以独立演进，不把所有逻辑耦合在单一脚本中。
- production build 可由 FastAPI 单端口提供，兼顾开发效率和交付稳定性。

**当前真实技术栈**：

- **Web 前端**：React 18、TypeScript、Vite、TanStack Query、Zustand。
- **服务端**：FastAPI、REST、WebSocket、Pydantic。
- **状态与数据**：SQLite 会话持久化、workspace 文件隔离。
- **数据处理**：pandas、openpyxl、pyarrow、matplotlib。
- **Agent 执行**：Pi runtime + typed spreadsheet tool service。

**视觉**：左右分区；左侧浏览器界面，右侧 FastAPI/Agent/Data 分层。只展示当前实际使用的技术路线，不介绍未使用框架。

**体现的能力**：技术选型、前后端边界、异步交互和工程可扩展性。

## 5. 技术架构：一次分析如何穿过整个系统

**页面标题**：每一次用户提问，都经过可观察、可约束的执行链路

**端到端链路**：

上传与校验 → 工作簿解析 → Agent 规划 → Python/表格工具执行 → 结果校验 → 报告与产物 → SQLite 历史恢复

**关键技术设计**：

- 前端通过 WebSocket 接收计划、步骤、reasoning、产物和终态事件。
- FastAPI 管理 conversation、run、artifact 和删除生命周期。
- Agent 只在会话 workspace 中读取数据、生成脚本和产物。
- SQLite 保存消息和运行终态，刷新或服务恢复后可以重新加载。
- Pi runtime 使用 context 隔离、工具限制和 `GIT_DIR=/dev/null` 阻断越权修改源码。

**视觉**：横向链路图；在 WebSocket、workspace、SQLite 三个关键接口处突出工程边界。

**体现的能力**：系统设计、状态机、异步任务、持久化和安全隔离。

## 7. 测试不是收尾动作，而是贯穿整个开发过程

**页面标题**：我把主要工程投入放在“如何证明它真的可靠”

**测试方法**：

- 每次关键问题先建立稳定复现，再做最小修复和回归测试。
- 数据解析、Planner、执行器、结果校验、API、WebSocket、Session 和 workspace 都有针对性测试。
- 新增能力不只验证 happy path，还覆盖取消、超时、错误输入、路径安全和状态恢复。
- 每次关键修改后重新执行全量测试和 production build，避免局部修复破坏其他链路。

**测试覆盖维度**：

1. **数据层**：复杂表头、重复列、行分组、公式表头、工作簿读取。
2. **Agent 层**：规划、prompt、执行、结果校验、工具约束。
3. **服务层**：REST、WebSocket、运行状态、取消、并发和会话删除。
4. **用户层**：上传、分析、追问、下载、刷新恢复和故障提示。

**视觉**：测试金字塔或四层覆盖图；从数据层一直连接到真实用户旅程。

## 9. 真实故障让我建立了完整的问题闭环

**页面标题**：成熟的工程能力，体现在问题能够被定位并防止复发

**案例一：Agent 越权执行 Git**

- 根因：数据分析 Agent 读取了仓库编码规则，把提交要求误用于分析任务。
- 修复：禁用 context files、限制工具、设置 `GIT_DIR=/dev/null`、补回归测试。

**案例二：运行时脚本触发服务 reload**

- 根因：开发热重载监控范围包含 Agent 生成的 workspace 脚本。
- 修复：稳定运行模式不 reload；开发模式只监控 `app/`。

**案例三：空白问题仍可提交**

- 根因：只检查文件，没有统一对 query 做 `trim()`。
- 修复：提交入口、mutation 和按钮状态统一加入空白守卫。

**问题处理闭环**：稳定复现 → 定位通用根因 → 最小修复 → 回归测试 → 全量 Gate → 记录到 release/Backlog

**体现的能力**：根因分析、风险隔离、最小变更和防复发机制。

## 11. 这次项目形成了一套可迁移的工程建设方法

**页面标题**：我学到的不是某一个工具，而是如何管理复杂项目的交付质量

**工程方法论**：

- **先做技术拆分**：用清晰模块和接口控制复杂度。
- **再做状态与边界**：让成功、失败、取消和恢复都有确定语义。
- **用 Git 管理变化**：分支隔离、PR 审查、版本冻结和 release 记录。
- **用测试管理风险**：从单元回归到真实 E2E，用证据替代主观判断。
- **用故障推动演进**：每次事故都沉淀成通用修复、回归测试和工程约束。

**与普通 Vibe Coding 作品的区别**：

- 不只展示 happy path，也建设错误处理和恢复机制。
- 不只保留最终代码，也保留完整的版本和决策历史。
- 不只依赖模型“看起来正确”，还用标准答案、benchmark 和真实浏览器验证。
- 不只面向自己使用，而是以可维护、可交付和未来生产化为设计方向。

**体现的能力**：独立技术判断、系统性执行、质量意识和成熟工程沟通。

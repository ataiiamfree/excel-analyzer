## 一、现有 ReAct 方案问题分析

### 1.1 测试数据现状

基于近期测试记录的统计结果：

| 场景 | 正常 | 异常 | 估算成功率 |
|------|------|------|-----------|
| 采购 | 20 | 10 | ~67% |
| 稽查 | 7 | 10 | ~41% |
| 业扩 | 5 | 3 | ~63% |
| 营销 | 8 | 1 | ~89% |

### 1.2 主要失败模式

| 失败类型 | 出现次数 | 严重程度 |
|---------|---------|---------|
| 中间过程没有输出工具调用 | 9 | 最高频 |
| 中间思考过程有误（幻觉） | 6 | 高 |
| 结果/数据格式有误 | 4 | 高 |
| 工具调用格式错误 | 2 | 中 |
| 输出终止 | 1 | 中 |
| 图片URL伪造 | 1 | 低 |

### 1.3 根本性架构问题

**问题一：ReAct 不适合数据分析场景**

Excel 数据分析本质上是半确定性流程（读文件 → 理解结构 → 写分析代码 → 执行 → 生成报告）。用 ReAct 意味着每一步都让 LLM 自由决策，每一步都可能出错。ReAct 循环越多，累积失败率越高：

> 5步 ReAct 每步 90% 成功率 → 整体成功率只有 59%。这就是为什么稽查场景（步骤多）成功率最低。

**问题二：压缩方案治标不治本，且引入新问题**

- **场景C 丢弃最老的工具结果**：但数据分析中最早的步骤（读取 Excel 结构、理解列名）恰恰是最基础的。丢掉后模型不知道数据长什么样，后续分析必然出错。
- **LLM 迭代压缩 AI 思考（最多3轮）**：用 LLM 压缩 LLM 的输出，本身又消耗 token 和延迟，且每次压缩都有信息损失。如果思考冗长到需要压缩3轮，说明 prompt 设计有问题，而不是需要更好的压缩器。
- **codeTruncationLimit = 1500 字**：对数据分析结果来说太激进了。关键数据点经常在截断中丢失。

**问题三：工具调用格式不稳定是 ReAct 固有缺陷**

靠模型生成特定格式文本来触发工具调用（而非原生 function calling），格式错误是必然会持续出现的。提示词优化只能降低概率，无法根治。

**问题四："36K可以，48K以上可能超出" 暴露窗口瓶颈**

说明模型上下文窗口本身不够用，压缩只是续命。一个稍微复杂的分析任务，ReAct 循环几轮就到 48K 了。

### 1.4 恶性循环

当前方案的致命问题不是压缩算法不够好，而是 ReAct + 文本解析工具调用 这个架构选择导致了：

> token 膨胀 → 需要压缩 → 压缩丢信息 → 分析出错 → 重试 → token 更多

这是一个恶性循环。压缩方案再怎么优化，也是在这个循环里打补丁。

## 二、业界产品架构与开源框架参考

### 2.1 ChatGPT Code Interpreter 的架构思路

ChatGPT Code Interpreter（现名 Advanced Data Analysis）是这个领域的标杆，核心架构**不是 ReAct**：

```
用户上传 Excel + 提问
       ↓
LLM 生成 Python 代码（一次生成完整脚本）
       ↓
沙箱执行代码 → 产出 DataFrame / 图表文件
       ↓
执行结果（stdout + 图表路径）返回 LLM
       ↓
LLM 生成自然语言报告（引用图表）
       ↓
如有错误 → LLM 看 traceback → 修正代码 → 重新执行
```

**关键点：中间数据从不进入对话上下文。** 代码在沙箱里跑，数据在沙箱内存里，LLM 只看到 print 输出和报错信息。所以根本不存在 token 爆掉的问题。

### 2.2 GitHub 高星开源框架

| 项目 | Stars | 核心思路 | 参考价值 |
|------|-------|----------|---------|
| **Open Interpreter** | 55k+ | LLM 生成代码 → 本地执行 → 结果回传 | 最接近 Code Interpreter 的开源实现 |
| **TaskWeaver**（微软）| 5k+ | **代码优先的分析 Agent**，专为数据分析设计，明确避开 ReAct 的 token 膨胀 | **最值得参考** |
| **PandasAI** | 15k+ | 自然语言 → pandas 代码 → 执行 | 轻量级，适合简单查询 |
| **LangGraph** | 8k+ | 有状态的 Agent 图，支持分步执行和检查点 | 如果需要复杂流程编排 |

**TaskWeaver 特别值得研究** —— 微软在论文里明确指出：ReAct 模式在数据分析场景下，因为中间结果膨胀导致不可控的 token 增长，所以他们设计了 code-first 架构。这和我们遇到的问题一模一样。

## 三、Token 爆掉是否不可避免

### 结论：不是。Token 爆掉只在一种架构下不可避免 —— 把中间数据塞进对话上下文。

**Code-First 架构下的 token 消耗：**
- 用户问题：~500 tokens
- 系统提示词：~2000 tokens
- LLM 生成的代码：~500 tokens
- 代码执行结果（只有 stdout/图表路径）：~200 tokens
- LLM 最终回答：~1000 tokens
- **总计：~4000 tokens**，哪怕分析的是百万行数据

**ReAct 架构下做同样的事：**
- 每一步 ReAct 循环：thought (~300) + action (~200) + observation (~1000+) = ~1500 tokens
- 5-8 步循环 = **7500-12000 tokens 仅中间过程**
- 还没算系统提示词和历史记录

**差距是数量级的。** 不是压缩能解决的。

## 四、快照意图过强导致结果不输出

这是一个经典的 **instruction conflict**（指令冲突）。Prompt 里同时告诉模型：

1. "把中间结果写入快照，只返回 snapshot_id"
2. "给用户生成完整的分析报告"

模型无法可靠地区分"这是中间结果该存快照"还是"这是最终结果该展示给用户"。尤其是压缩后上下文丢失时，模型更容易搞混。

这个问题在 ReAct 框架内很难干净地解决，因为模型在"生成中间产物"和"生成最终产物"时用的是同一个输出通道。在 Code-First 架构中，这个问题天然不存在 —— 代码的 print() 输出就是给 LLM 看的摘要，文件输出就是给用户下载的数据，两个通道天然分离。

## 五、替代方案：Code-First Pipeline

### 5.1 核心理念

1. **LLM 只做两件事**：看数据结构写代码、看执行结果写报告
2. **数据永远不进 LLM**：百万行 Excel 在沙箱里处理，LLM 只看到 schema + 样本
3. **确定性流水线**：不用 ReAct，每一步输入输出固定，可预测可调试

### 5.2 四阶段流程

```
┌──────────────────────────────────────────────────────────┐
│  Stage 1: Profiler（数据画像，零 LLM 调用）                │
│  读取 Excel → 提取列名、类型、统计信息、样本行              │
│  输出 data_profile（~500 tokens），无论原始数据多大          │
└──────────────────────┬───────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────┐
│  Stage 2: Planner（代码生成，1 次 LLM 调用）               │
│  输入: system_prompt + data_profile + 用户问题              │
│  输出: 完整的 Python 分析脚本                               │
│  Token: ~3000 输入 + ~800 输出                             │
└──────────────────────┬───────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────┐
│  Stage 3: Executor（沙箱执行，零 LLM 调用）                │
│  执行代码 → 收集 stdout + 图表文件 + 导出文件               │
│  失败 → 自愈（LLM 看 traceback 修代码，最多重试 2 次）      │
└──────────────────────┬───────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────┐
│  Stage 4: Reporter（报告生成，1 次 LLM 调用）               │
│  输入: 用户问题 + stdout 摘要 + 图表路径                    │
│  输出: 自然语言分析报告                                     │
│  Token: ~2000 输入 + ~1500 输出                            │
└──────────────────────────────────────────────────────────┘
```

### 5.3 Token 消耗对比

| 场景 | ReAct 方案（当前） | Code-First 方案 |
|------|-------------------|----------------|
| 简单查询 | ~15K-25K | ~4K |
| 中等分析 | ~30K-48K | ~5K |
| 复杂分析 | ~48K+（爆掉） | ~7K |
| 失败重试 | 整个上下文翻倍 | +2K per retry |

### 5.4 现有组件复用

| 现有组件 | 在新架构中 | 备注 |
|----------|-----------|------|
| Python 沙箱执行 | **直接复用** | Stage 3 |
| 快照机制 | **不需要了** | 数据不进上下文 |
| 压缩算法 | **不需要了** | Token 总量 <8K |
| 工具调用解析 | **不需要了** | 只有代码生成和报告生成 |
| 系统提示词 | **大幅简化** | 不需要 ReAct 指令 |

### 5.5 模型选择

| 模型 | 参数量 | 用途 | 理由 |
|------|--------|------|------|
| **Qwen2.5-Coder-32B** | 32B | 代码生成 | 中国开源模型中代码能力最强 |
| **Qwen2.5-72B** 或 **DeepSeek-V3** | 72B/671B | 报告生成（可选） | 中文写作能力更强 |

部署方式：vLLM 或 Ollama 本地部署，不依赖任何外部 API。

## 六、大数据量处理：行多与列多

### 6.1 行多（上万行）：根本不是问题

在 Code-First 架构里，数据行**永远不进入 LLM**。

```
LLM 看到的：             沙箱里实际处理的：

"shape": [50000, 12]  →  df = pd.read_excel(...)  # 5万行全在内存里
"sample": [前3行]      →  df.groupby(...).sum()     # pandas 直接算
```

哪怕 100 万行，LLM 看到的也只是一个数字 `"rows": 1000000`。**行数跟 token 没有任何关系。**

### 6.2 列多（上百个字段）：分层画像解决

100 个列全量画像 ≈ 5000 tokens，需要优化。方案：**分层画像 + 列名分组**。

**Layer 1: 全量概览（所有列，极简信息）**
```
列名        类型       空值率
─────────  ────────   ──────
采购日期    datetime   0%
采购金额    float64    0%
供应商名称   object     2%
...（100列，每列仅 ~10 tokens）
总计：~1000 tokens
```

**Layer 2: 重点列详情（只展开相关列）**
```
采购金额:
  min=150, max=98000, mean=12350
  sample: [1500, 23000, 8700]
采购日期:
  range: 2024-01-01 ~ 2025-12-31
（只展开 10-15 个相关列，~800 tokens）
```

**列名分组压缩（处理重复模式）：**
```python
# 原始：36 列逐一列出
"1月_计划量", "2月_计划量", ... "12月_计划量",   # 12列
"1月_实际量", "2月_实际量", ... "12月_实际量",   # 12列
"1月_完成率", "2月_完成率", ... "12月_完成率",   # 12列

# 分组后：3 行描述
"[1-12月]_计划量 (12列, float64)"
"[1-12月]_实际量 (12列, float64)"
"[1-12月]_完成率 (12列, float64)"
```

**效果：**

| 场景 | 原始 profile tokens | 优化后 tokens |
|------|-------------------|--------------|
| 10 列 | ~500 | ~500（不需要优化） |
| 50 列 | ~2500 | ~1200 |
| 100 列 | ~5000 | ~1800 |
| 200 列 | ~10000 | ~2200 |

**LLM 拿到精简 profile 后，生成的代码依然可以操作所有列**，因为代码可以写循环：

```python
month_cols = [f"{m}月_计划量" for m in range(1, 13)]
df[month_cols].sum()  # 完全正确，不需要 LLM "看到" 每一列的数据
```

## 七、明细数据输出的处理方式

### 7.1 核心思路：双通道输出

明细数据根本不应该经过 LLM。

```
                    代码在沙箱里执行
                         │
              ┌──────────┴──────────┐
              ▼                      ▼
        stdout（文字）            文件（数据）
        给 LLM 写报告             直接给用户下载
              │                      │
         限制在 2K 内             无大小限制
              │                      │
              ▼                      ▼
     "共筛选出 237 条记录，       result.xlsx
      金额范围 10万~98万..."      （完整明细）
```

### 7.2 代码生成规则

Planner 的 prompt 里加一条规则：

> 结果超过 20 行就写文件（to_excel），不要 print 全部内容。用 print 输出摘要信息。

LLM 生成的代码自然变成：

```python
result = df[df["采购金额"] > 100000]

# 明细 → 写文件（不占 token）
result.to_excel("output/筛选结果.xlsx", index=False)

# 摘要 → print（给 LLM 写报告用）
print(f"筛选完成：共 {len(result)} 条记录")
print(f"金额范围：{result['采购金额'].min():,.0f} ~ {result['采购金额'].max():,.0f}")
print(f"\n前5条预览：")
print(result.head().to_markdown())
```

### 7.3 前端展示

```
┌─────────────────────────────────────────────┐
│  分析结果                                    │
│                                             │
│  已筛选出采购金额超过10万的明细记录，          │
│  共 237 条，金额范围 10.2万 ~ 98.0万。        │
│                                             │
│  预览（前 20 行）：                           │
│  ┌──────┬────────┬──────────┬────────┐      │
│  │ 序号 │ 日期    │ 供应商    │ 金额   │      │
│  │ 1    │ 01-03  │ XX公司   │ 98,000 │      │
│  │ ...  │        │          │        │      │
│  └──────┴────────┴──────────┴────────┘      │
│                                             │
│  [下载完整明细 (237条)]                       │
└─────────────────────────────────────────────┘
```

## 八、多步任务的数据传递

### 8.1 单次任务内的多步

**数据留在沙箱里，代码之间通过文件/变量传递，LLM 只传"做什么"，不传数据本身。**

当任务可以一次性生成代码时，步骤间通过 Python 变量传递：

```python
# LLM 一次生成的完整脚本

# Step 1: 按供应商汇总
supplier_summary = df.groupby("供应商")["采购金额"].agg(["sum","count"])
top10 = supplier_summary.head(10)

# Step 2: 直接用 Step1 的变量
top10_names = top10.index.tolist()
detail = df[df["供应商"].isin(top10_names)]
```

### 8.2 多轮对话的数据传递

用户追问时，中间结果通过文件持久化：

```
第 1 轮：
  代码执行 → result.to_pickle("output/supplier_summary.pkl")
  LLM 只看到 → "共 45 家供应商，前三名是..."

第 2 轮：
  LLM 收到 →  profile + 上一轮摘要 + 文件列表
              ["output/supplier_summary.pkl"]
  代码生成 → summary = pd.read_pickle("output/supplier_summary.pkl")
```

LLM 在第 2 轮看到的全部信息（~2500 tokens）：

```
## 数据概况
{原始 profile}

## 历史对话
用户：帮我按供应商汇总
助手：共 45 家供应商，前三名为 A公司(320万)、B公司(280万)...

## 当前工作目录已有文件
- upload/采购数据.xlsx
- output/supplier_summary.pkl（供应商汇总结果）

## 当前问题
把排名前5的供应商拉出月度趋势
```

**哪怕数据有百万行、上百列，LLM 看到的永远就这么多。**

### 8.3 本质区别

| | ReAct | Code-First |
|---|---|---|
| 步骤间数据传递 | 塞进 LLM 上下文 | 沙箱内文件/变量 |
| 10万行中间结果 | → token 爆掉 | `df.to_pickle()` → 0 tokens |
| 多轮追问 | 累积所有历史 | 只保留摘要 + 文件列表 |
| token 随轮次增长 | 线性/指数增长 | **恒定 ~3K-4K** |

## 九、Plan-Execute vs ReAct 对比

### 9.1 "ReAct 有 double check，更严谨" —— Plan-Execute 同样可以

Plan-Execute 同样具备自我修正能力，区别只是上下文管理方式不同：

```
执行步骤 → 检查结果
              │
    ┌─────────┼──────────┐
    ▼         ▼          ▼
  成功      代码报错    结果不对
  │         │            │
  │     ┌───▼────┐   ┌───▼──────┐
  │     │ Repair │   │ Re-plan  │
  │     │ 修代码  │   │ 调整计划  │
  │     └───┬────┘   └───┬──────┘
  ▼         ▼            ▼
 下一步   重新执行     从当前步继续
```

**每一次 check / repair / re-plan 都是独立的 LLM 调用，不往上下文里追加。** 修正能力一样，但上下文不膨胀。

### 9.2 "ReAct 更灵活" —— 取决于场景

```
场景光谱：

开放探索型                                目标明确型
←───────────────────────────────────────────→
"帮我调研竞品"                "根据采购台账分析采购时长"

  适合 ReAct                    适合 Plan-Execute
  不知道要几步                    步骤基本可预知
  不知道用什么工具                工具就那几个
  不知道何时完成                  交付物很明确
```

在数据分析场景中，ReAct 的"自由探索"不是优势，反而是负担：

| ReAct 的"灵活" | 在数据分析场景中的实际表现 |
|---|---|
| 自由决定下一步 | 有时候决定不调工具（头号 bug） |
| 可以随时改变方向 | 改着改着把前面的上下文挤掉了 |
| 不需要预先规划 | 导致走弯路、重复计算 |
| 每步都能反思 | 反思本身消耗大量 token |

### 9.3 Plan 不是死板的

Plan 可以在执行过程中动态修正：

```
Plan-Execute ≠ 一次规划，死板执行
Plan-Execute = 规划 → 执行 → 检查 → 必要时重新规划
```

即使用户问题很开放（如"帮我看看这数据有什么有趣的发现"），Plan 阶段本身就能处理：

```json
{
  "steps": [
    {"tool": "python", "task": "探索性分析：描述性统计、相关性、分布、异常值检测"},
    {"tool": "python", "task": "基于EDA结果深入分析"},
    {"tool": "report", "task": "将发现整理成报告"}
  ]
}
```

**开放性在 Plan 阶段就被 LLM 转化成了具体步骤。** 跟人类分析师一样 —— 拿到模糊需求，先想做哪几件事，然后一件件做。

### 9.4 "Agent + 多工具" 不等于 "必须用 ReAct"

```
Agent = LLM + Tools + 编排策略
                       ↑
                  ReAct 只是其中一种选择

ReAct:        一个不断追加的长对话，每次带全部历史
Plan-Execute: 多个独立的短调用，每次只带当前步骤所需信息
```

```
ReAct 的上下文（一根不断变长的线）：

tokens
  ▲                                          / 爆了
  │                                       /
  │                                    /
  │                                 /
  │                              /
  │                           /
  │                        /
  │                     /
  │                  /
  │               /
  │            /
  │         /
  │      /
  └──────────────────────────────────────▶ 步骤


Plan-Execute 的上下文（独立的短柱子）：

tokens
  ▲
  │
  │  ┌──┐       ┌──┐       ┌──┐       ┌──┐
  │  │3K│       │3K│       │3K│       │3K│    ← 每次都一样大
  │  │  │       │  │       │  │       │  │
  └──┴──┴───────┴──┴───────┴──┴───────┴──┴──▶ 步骤
```

## 十、知识库（RAG）的集成方式

### 10.1 架构上的解耦

知识检索是一个可插拔的工具，和核心流程解耦：

```
现在只做 ChatExcel：
  Orchestrator
    ├── Profiler       [用]
    ├── Planner        [用]
    ├── PythonSandbox  [用]
    ├── Knowledge      [不注册]  ← Planner 不会生成 knowledge 步骤
    └── Reporter       [用]

后续加上 RAG：
  Orchestrator
    ├── Profiler       [用]
    ├── Planner        [用]  ← prompt 加一句"可用知识检索工具"
    ├── PythonSandbox  [用]
    ├── Knowledge      [注册]
    └── Reporter       [用]  ← 报告生成时多一个知识片段输入
```

核心代码（Orchestrator、TaskContext、PromptAssembler）一行不用改。

### 10.2 复合任务处理（Excel + 知识库 + 长报告）

以"根据采购台账分析采购时长，生成8000字报告"为例：

```
Task Planner → 生成计划，含 python 和 knowledge 两类步骤
     │
     ├── Data Pipeline（Code-First）：计算统计、生成图表
     │     └── 产出：stdout 摘要 + 图表文件
     │
     ├── Knowledge Retriever（RAG）：检索行业规范、管理制度
     │     └── 产出：知识片段 + 来源标注
     │
     └── Report Generator（分章节生成）：
           └── 每章独立 LLM 调用，输入该章数据 + 知识
           └── 7 章 × ~3K tokens/章 = ~21K（分散在 7 次调用中）
```

### 10.3 8000 字报告的生成策略

不要一次生成 —— 分章节，每章独立调用：

```
每章的 LLM 输入（~3K tokens）：
- 完整报告大纲（保持连贯）
- 当前章节标题和字数要求
- 该章节对应的数据摘要
- 该章节对应的知识片段 + 来源
- 上一章结尾 200 字（衔接用）
```

## 十一、完整工程设计

### 11.1 TaskContext —— 上下文保护机制

整个架构的核心。解决"步骤间需要传递信息，但不能无限累积"的问题。

```python
@dataclass
class TaskContext:
    # 固定区（不增长）
    task_id: str
    user_query: str              # 用户原始问题
    data_profile: dict           # 数据画像（固定大小）
    plan: ExecutionPlan          # 执行计划

    # 摘要区（有严格上限）
    step_summaries: OrderedDict  # 每步摘要，每条限 200 字
    key_findings: list           # 关键发现，限 10 条

    # 文件区（只存文件名，不存内容）
    workspace_files: list

    # Token 预算
    BUDGET = {
        "user_query":      200,
        "data_profile":    800,
        "plan_overview":   300,
        "step_summaries":  600,   # 总和上限
        "key_findings":    200,
        "workspace_files": 200,
        "step_instruction":300,
        # 总计上限:       2600 tokens
    }
```

**摘要区增长控制**：
- 每步 stdout 用规则提取摘要（不调 LLM），限 200 字
- 总摘要超 600 tokens → 最老的几条合并为一句话
- 关键发现只保留最近 10 条

**Prompt 组装**：
- 每步的 Prompt 从 TaskContext 中选取所需信息组装
- 安全兜底：超 4K tokens → 循环压缩摘要区直到达标
- **任何一次 LLM 调用的输入恒定 ≤ 4K tokens**

### 11.2 三级错误处理

```
Level 1: Repair（代码修复）
  触发：Python 执行报错
  处理：traceback + 原代码 + profile → LLM 修正
  重试：最多 2 次，每次独立调用

Level 2: Re-plan（重新规划）
  触发：修复 2 次仍失败
  处理：Planner 从失败步开始重新规划

Level 3: Graceful Degradation（降级）
  触发：重新规划后仍失败
  处理：跳过该步骤，报告中标注"未能完成"
```

### 11.3 全任务 Token 估算

```
简单任务（纯 Excel 分析）：
  Planner ×1 + CodeGen ×3 + Repair(概率) ×1
  总计: ~24K tokens，分散在 5 次独立调用中

复杂任务（Excel + 知识库 + 8000字报告）：
  Planner ×1 + CodeGen ×5 + Repair ×2 + Report ×7
  总计: ~74K tokens，分散在 18 次独立调用中
  单次最大 5.5K tokens
```

### 11.6 所有痛点的对应解决

| 当前痛点 | 新方案如何解决 |
|---------|--------------|
| Token 爆掉 | 全流程 <8K tokens，不可能爆 |
| 工具调用格式不稳定 | 没有工具调用，只有代码生成 |
| 中间过程不调用工具 | 没有中间过程，代码一次生成 |
| 快照意图太强不输出 | 没有快照，stdout 和文件天然分离 |
| 图片URL幻觉 | 图表是代码生成的真实文件 |
| 压缩导致信息丢失 | 不需要压缩 |
| 输出终止 | 单次调用短小，极少触发 |
| 成功率低 | 2-3 次确定性调用 vs 5-8 次概率性循环 |
| 调试困难 | 每步独立，可单独重跑和检查 |

## 附：与 ReAct 的最终对比

| 维度 | ReAct（当前） | Plan-Execute（建议） |
|------|-------------|---------------------|
| 上下文管理 | 单一对话累积，需要压缩 | TaskContext 结构化，有预算 |
| 单次调用大小 | 随步骤增长，不可预测 | 恒定 ≤ 4K tokens |
| 总 token 消耗 | 指数级增长 | 线性（次数 x 固定大小） |
| 自愈能力 | 有，但在同一对话中追加 | 有，独立调用不污染上下文 |
| 灵活性 | 每步自由决策 | Plan 阶段灵活，执行阶段精确 |
| 可调试性 | 难（长对话难定位） | 易（每步独立可重跑） |
| 失败影响 | 一步失败可能污染后续 | 一步失败不影响已完成步骤 |
| 信息丢失 | 压缩导致不可预测的丢失 | 明确的保留/丢弃策略 |
